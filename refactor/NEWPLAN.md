> **Статус**: утверждён к исполнению. Заменяет PLAN.md (первая попытка, вердикт пользователя — регрессия, см. `handoffs/refactor-partial-split-2026-07.md`).
>
> **Отправная точка**: `refactor/original/BorderlessWindow.cs` (5545 строк) — единственный рабочий baseline.
> Текущие `Controls/*.cs` — архив неудачной попытки; за основу НЕ брать, можно подглядывать только имена файлов/границы разбиения.

## 0. Цель и НЕ-цель

**Цель**: получить те же 7 partial-файлов в `Controls/`, поведение которых **бит-в-бит совпадает** с монолитом, минус:
1. мёртвый код (доказуемо недостижимый / ни разу не используемый);
2. код для тестирования (диагностическое логирование TsLog/SnapLog, debug-логи грипов).

**НЕ-цель** (запрещено в рамках этого плана):
- НИКАКИХ поведенческих фиксов (ни «фикс 1», ни «фикс 2» из прошлой попытки, ни новых);
- НИКАКОГО удаления ControlzEx — он несущий (урок №3 handoff'а). Флаги `UseThemedSystemFrame` / `AttachControlzExChrome` и оба пути под ними переносятся как есть;
- НИКАКОГО «улучшения» кода при переносе: не переименовывать, не менять сигнатуры, не схлопывать ветки, не трогать комментарии (кроме удаляемой диагностики);
- НЕ считать мёртвым кодом ветки под `const bool = false` (`IncludeCaptionForSnap`, `DisableDwmNcRendering` и т.п.) — это конфигурация, охраняющая живые альтернативные пути.

## 1. Уроки прошлой попытки, встроенные в план

| Урок (handoff) | Как учтён |
|---|---|
| Big-bang сделал регрессии неразделимыми | Каждый этап = отдельный коммит + сборка + ручной smoke-тест пользователя. Следующий этап не начинается без вердикта «OK» |
| Статическая верификация ≠ работоспособность | Единственный критерий приёмки этапа — сборка и ручной тест пользователем на Windows |

| Диагностику удалили до приёмки поведения | Диагностика удаляется ПОСЛЕДНЕЙ (этап 4), только после того, как разбиение принято пользователем как эквивалентное |
| Фиксы «по симметрии» без доказанного механизма | Фиксов нет вообще |

## 2. Этапы

### Этап N0 — Baseline и инфраструктура (без изменений кода)

1. Пользователь собирает проект из `original/BorderlessWindow.cs` и подтверждает: поведение = эталон (все известные артефакты исходника — призрак при ресайзе и т.д. — считаются частью эталона, их НЕ чиним).
2. (Опционально, но желательно) добавить в `refactor/` минимальный тестовый `.csproj` (`net10.0-windows`, WPF, PackageReference на ControlzEx + Microsoft.Xaml.Behaviors, `<Compile Include="ControlsV2/**/*.cs"/>`), чтобы агенты могли гонять `dotnet build` как smoke-check. Если среды нет — сборку делает только пользователь, это фиксируется в TASKS.
3. Новые файлы кладём в **`refactor/ControlsV2/`**, чтобы не перезаписать архив первой попытки. Переименование в `Controls/` — отдельным финальным шагом после приёмки.

**Приёмка**: пользователь подтвердил эталонное поведение. Коммит.

### Этап N1 — Механическое разбиение, MOVE-ONLY (ноль изменённых строк)

Разбить монолит на 7 partial-файлов **только перемещением строк**. Ни одна строка кода или комментария не меняется, не удаляется и не добавляется — кроме обязательного «клея»:
- шапка `using ...; namespace ControlPanel { public partial class BorderlessWindow ... }` в каждом файле (`: Window` и class-level XML-doc — только в CORE);
- ключевое слово `partial`;
- полный набор `using` копируется во все файлы (неиспользуемые - предупреждение, не ошибка; чистка — на этапе N4, не раньше).

Карта разбиения (по фактическим регионам монолита, границы уточняются по коду, а не по прошлой попытке):

| Файл | Содержимое (строки монолита ориентировочно) |
|---|---|
| `BorderlessWindow.cs` (CORE) | Класс-декларация, конструктор, все const-флаги конфигурации, общие поля, OnSourceInitialized/WndProc-диспетчер |
| `BorderlessWindow.Chrome.cs` | ControlzEx attach, themed frame, DWM-атрибуты, NCCALCSIZE/hit-test |
| `BorderlessWindow.Taskbar.cs` | Auto-hide панель: edge watcher, shrink/expand, регион «Динамическое освобождение нижнего края» |
| `BorderlessWindow.SnapResize.cs` | Регион «Snap joint-resize follow (БАГ 1)» (~2537–3488), free-edge grip, divider guide |
| `BorderlessWindow.Unsnap.cs` | Регион «Caption un-snap restore drag (ЗАДАЧА 3)» (~3592–4167) |
| `BorderlessWindow.Animation.cs` | Регион «Стартовое появление (маска + гашение вспышки)» (~4346–4735), «Маскировка зазора» |
| `BorderlessWindow.Interop.cs` | Регион «WinAPI interop» (~5228–5544), «Сохранение положения окна» |

**ВАЖНО про диагностику на этом этапе**: регион «Диагностика Snap» (~4207–4314), `TsLog`/`SnapLog` вызовы (~78 вхождений), `GripDebugLog`, `EnableTroubleshootLog` — **переносятся как есть**. Они понадобятся для отладки, если разбиение что-то сломает.

**Механическая верификация перед передачей пользователю** (необходимое, но НЕ достаточное условие):
- скрипт «учёта строк»: каждая непустая строка монолита обязана присутствовать ровно в одном из 7 файлов (нормализация — только trim); допустимые расхождения — только строки «клея» из явного белого списка;
- баланс `{}`/`()`/`[]` по каждому файлу (с вырезанием строк/комментариев);
- `partial class BorderlessWindow : Window` ровно один раз; дубли const/полей между файлами = 0 (урок CS0102).

**Приёмка**: пользователь собирает и гоняет полный чек-лист поведения (см. §3). Вердикт «эквивалентно эталону» — или откат/починка ЭТОГО этапа до перехода дальше. Коммит.

### Этап N2 — Удаление доказуемо мёртвого кода (по одному члену за коммит)

«Мёртвый код» определяется ТОЛЬКО так: приватный член (метод/поле/константа), на который **компилятор/IDE не находит ни одной ссылки** во всех 7 файлах (предупреждения CS0169/IDE0051 или ручная проверка «0 ссылок»). Grep-эвристики недостаточно.

Правила:
- по одному члену (или тесно связанной группе) за коммит, с указанием в сообщении коммита, почему член мёртв;
- ветки под const-флагами НЕ считаются мёртвыми (см. НЕ-цели);
- после каждой пачки удалений — сборка; после всего этапа — ручной smoke-тест пользователя.

**Приёмка**: сборка чистая, smoke-тест «эквивалентно». Коммит(ы).

### Этап N3 — Удаление тестового/диагностического кода (последним!)

Только после того, как пользователь принял N1+N2 как эквивалент эталона:
1. Сначала — выключение: `EnableTroubleshootLog = false`, `GripDebugLog` уже false. Сборка + smoke-тест (ловим случайные side-effects в вызовах логгера).
2. Затем — физическое удаление: регион «Диагностика Snap», все вызовы `TsLog(...)`/`SnapLog(...)` (удалять только statement целиком; если вызов — единственное тело ветки, оставить пустую ветку или скобки как есть — не «оптимизировать» условие), сами методы-логгеры и их поля.
3. Один подсистемный файл = один коммит (7 коммитов максимум).

**Приёмка**: сборка + полный чек-лист §3. Коммит(ы).

### Этап N4 — Финализация

1. Чистка неиспользуемых `using` в каждом файле (по предупреждениям компилятора).
2. Переименование `ControlsV2/` → `Controls/` (старый архив — в `Controls-failed-2026-07/` или удалить по решению пользователя).
3. Обновить TASKS.md/AGENT_LOG.md, закрыть план.

**Приёмка**: финальная сборка + полный чек-лист §3.

## 3. Чек-лист ручной приёмки (пользователь, Windows)

Прогоняется на этапах N1, N3, N4 (на N2 — сокращённый smoke):
1. Запуск приложения: нет белой полосы слева, нет вспышек, стартовая маска работает.
2. Обычный ресайз за края/углы: поведение призрака НЕ ХУЖЕ эталона (эталон = монолит, его артефакты допустимы).
3. Maximize/restore, поведение с auto-hide панелью задач (выезд без лага, окно не дёргается).
4. Snap 2×2 / 2×1, драг общего разделителя (joint-resize follow).
5. Стягивание за шапку из snap (un-snap restore drag) — включая полноэкранный/maximized случай.
6. Стягивание за шапку из fullscreen/maximized — работает (регрессия первой попытки!).
7. Сохранение/восстановление положения окна между запусками.
8. При наличии второго монитора: пункты 2, 4, 5 на стыке мониторов с разным DPI.

Любой пункт «хуже эталона» = стоп, откат последнего этапа, разбор с диагностикой (она ещё в коде до N3).

## 4. Роли и дисциплина

- Исполнитель этапа пишет отчёт в AGENT_LOG.md (факты и числа, не «должно работать»).
- Аудитор проверяет ТОЛЬКО механику (учёт строк, баланс, дубли) и НЕ предлагает поведенческих правок.
- Единственный, кто выносит вердикт «эквивалентно/регрессия» — пользователь, после реальной сборки и теста.
- Все идеи улучшений (включая фиксы призрака/щели) — только в IDEAS.md, реализация — отдельным проектом ПОСЛЕ приёмки этого плана.
