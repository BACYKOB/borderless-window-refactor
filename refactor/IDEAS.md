> Создан аудитором (Fable) 2026-07-05 по итогам ревью T3/T4. T6 ДОПОЛНЯЕТ этот файл (находки собственной верификации + накопленное T5), существующие пункты не удаляет. Ничего из этого файла НЕ реализовывать до окончания T6 и ручного тестирования пользователем.

## 1. Фикс 2 (левый ghost-guard) — эскалации и пограничные случаи

- **Динамический inset 2px на время left-drag** (эскалация из PLAN Часть 4): если 1px-guard не убьёт призрак полностью на стенде пользователя (4K @150%), поднять inset до 2px только при активном `_divDragging`/`_sizingEdge` слева, со снятием через `SWP_FRAMECHANGED` на отпускании. Не включать превентивно — каждый лишний px съедает клиент.
- **Кламп у левого края экрана перекрывает guard** (аудит): в `ThemedFrameNcCalcSize` кламп `calc.rgrc0.Left = wa.Left` (зона `EdgeClampMaxOverhang` = 32px выступа) перезаписывает inset → на последних ~32px тяги к левому краю монитора guard исчезает, призрак может мигнуть. У верха то же самое, но верхний край в этой зоне не движется, а левый — движется. Возможное решение: применять inset ПОСЛЕ клампа (клампить видимый край, потом инсетить client внутрь) — потребует пересверки с `ApplySnapLayoutGapFix`. Делать только если пользователь заметит мигание у края.
- **Правый край**: у правого края guard'а нет (client full-bleed справа). Призрак исторически репортился слева/сверху, но если проявится справа — рецепт готов: зеркальный `rgrc0.Right -= ThemedFrameInset`. Не делать превентивно.
- **DWM-полоса при snapped = 2px боковой границы — желаемое поведение** (подтверждено пользователем: floating 1px, snap 2px). Если когда-нибудь захочется 1px и при снапе: вариант — при снапе с внутренним разделителем обнулять `BorderThickness.Left` И красить DWM-полосу (сейчас DWM-полоса красится ПОВЕРХ WPF-линии, визуально сливаясь в 2px).

## 2. Фикс 1 (seam-gap) — оптимизации и эскалации

- **Пропуск no-op SetWindowPos в grower-first pre-pass** (аудит): `FollowFrameResizeNeighborsPre` шлёт соседу SetWindowPos на каждый WM_WINDOWPOSCHANGING, даже если предложенный край не изменился с прошлого коммита — лишний кросс-процессный round-trip (геометрически no-op, но не бесплатный). Лечится полем «последний закоммиченный край на соседа» (Dictionary hwnd→int), сброс при `WM_EXITSIZEMOVE`.
- **Инвалидация MINMAXINFO-кэша по времени**: `_divMinTrackCache` живёт весь драг. Если чужое окно меняет min-size ВО ВРЕМЯ драга (экзотика), кламп устареет — докрывается `ArmDivReleaseRealign`. При желании: TTL ~500ms на запись вместо жизни-до-релиза.
- **Кэш и для fallback-пути**: fallback (двухпроходная схема) не использует `TryGetMinTrackSize` — ему кэш не нужен. Но если fallback когда-то захочется ускорить, тот же кэш применим к чтению `TryGetVisibleBounds` (сейчас 2-4 вызова на соседа на кадр — DWM-запрос тоже не бесплатный, хоть и внутрипроцессный).
- **Транзитивная 4-оконная группа** (из PLAN): co-tiles сейчас захватываются только в одной колонке/строке от нас. Сетка 2×2 при драге центрального перекрестья двигает 3 окна из 4; четвёртое (по диагонали) не следует. Решение — транзитивное замыкание flush-соседства при захвате. Отложено: редкий сценарий, рост сложности заметный.
- **Coalescing mouse-move до кадра**: сейчас `UpdateDividerJointResize` дергается на каждый WM_MOUSEMOVE; при 1000Hz-мыши это больше batch'ей, чем кадров DWM. Вариант: копить последний dividerX и коммитить в `CompositionTarget.Rendering` (осторожно: похожий эксперимент `EnableDividerFrameSync` был признан тупиком для СИНХРОНИЗАЦИИ, но как чистый rate-limiter может быть безопасен).

## 3. Гигиена проекта

- **Удалить ControlzEx PackageReference из csproj** — руками пользователя после T6 (кода ControlzEx в partial'ах уже 0 вхождений).
- **Вынос const-флагов в настройки/статический конфиг**: сейчас откат любого фикса = правка константы + перекомпиляция. Если пользователь захочет A/B-переключение без пересборки — читать флаги из `app.config`/JSON один раз при старте (сделать `static readonly` вместо `const`; JIT всё равно свернёт ветки после первого чтения).
- **Артефакты сверки в приёмку** (процессный урок аудита): критерии вида «fallback бит-в-бит» подкреплять приложенным diff-выхлопом по диапазонам строк оригинала, а не словом «сверено». T6: включить фактические грэп-выхлопы в сдачу (уже в приёмке T6).
- **Ветки git**: T3 уходила через PR #3 до появления правила «всё напрямую в main». T6 проверить: все рабочие ветки слиты, расхождений с main нет.

## 4. Идеи аудита на будущее (после ручных тестов)

- **Телеметрия кадров драга (dev-only)**: временный счётчик «batch'ей на кадр DWM» (через `DwmFlush`/`CompositionTarget.Rendering`) для объективной проверки «ровно один коммит за кадр» на стенде пользователя — вместо субъективного «зазора не видно». Удалить после проверки.
- **Проверка на 2+ мониторах с разным DPI**: `ThemedFrameInset` — физ. px, DIP-пересчёт бордера на per-monitor DPI меняется в `OnDpiChanged`; сценарий «драг разделителя на стыке мониторов с разным масштабом» в 7 ручных тестов не входит — добавить восьмым.
- **`GetSystemMetrics(SM_CXDRAG)`-зона вместо константного 32px `EdgeClampMaxOverhang`** — косметика, низкий приоритет.
