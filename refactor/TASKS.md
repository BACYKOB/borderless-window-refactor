# TASKS — поэтапные задачи рефакторинга BorderlessWindow

> Каждую задачу закрывает отдельный агент-исполнитель. Работу проверяет агент-ревьюер (см. «Процесс» внизу).
> Перед началом ЛЮБОЙ задачи прочитай: `AGENT_LOG.md` → `PLAN.md` → раздел своей задачи здесь. Больше ничего читать целиком не нужно — точечные чтения по указателям из METHOD_MAP.
> Статусы: `TODO` / `IN PROGRESS (кто)` / `REVIEW` / `DONE`. Исполнитель обновляет статус и добавляет запись в `AGENT_LOG.md` (раздел «Журнал сессий») при старте и при сдаче.

## T4 — SnapResize + фикс зазора (`BorderlessWindow.SnapResize.cs`) — `DONE` (принята с правкой аудитора)

**Модель**: Fable. **Ревью/правка**: аудитор (Fable), 2026-07-05 — добавлен кэш MINMAXINFO на время драга (`_divMinTrackCache`): без него `TryGetMinTrackSize` слал синхронный кросс-процессный запрос (таймаут 50ms) каждому соседу на каждый mouse-move кадр — перф-риск (рывки). См. журнал.

**Вход**: METHOD_MAP.md; PLAN.md Часть 3; `handoffs/snap-joint-resize.md`, `handoffs/resize-jitter.md`; лог бага `logs/snap-joint-resize-gap.log`.

**Сделать**: перенос snap-методов + внедрить `EnableSeamGapFix`:
1. DivGrip-путь: вместо предиктивного реального сдвига соседей (`ApplyDividerBatch(moveOur=false)` + финальный) — предвычисление клампа `effDiv` через `WM_GETMINMAXINFO` соседа и ОДИН атомарный `BeginDefer/EndDeferWindowPos` (мы + соседи + co-tiles). Существующий realign на отпускании (`ArmDivReleaseRealign`) сохранить.
2. Frame-resize путь: «grower-first» — когда сосед растёт, коммитить его новый край в `WM_WINDOWPOSCHANGING` (до нашего кадра); когда сосед сжимается — как сейчас, после `WM_WINDOWPOSCHANGED`. Инвариант: переходное состояние всегда перекрытие, никогда зазор.
3. При `EnableSeamGapFix=false` — прежняя двухпроходная схема без изменений.

**Границы**: PassiveFollow, tile-group, free-edge grip, snap-follow таймер — пе��енос 1:1 без изменений; `CaptionUnsnapRestoreThresholdDip` не трогать (он в Unsnap).

**Приёмка**: fallback идентичен исходнику; в новом пути наше окно двигается ровно один раз за кадр (нет двойного move → нет ghost-trail регрессии); min-size clamp обработан ДО коммита; обработчик в `WM_WINDOWPOSCHANGING` не конфликтует с существующей pending-rect логикой ядра.

## Процесс

- **Модель**: каждую задачу выполняет модель, указанная в её заголовке (см. таблицу «Назначение моделей»). Пользователь запускает нужную модель на нужную задачу.
- **Исполнитель**: берёт ОДНУ задачу в статусе `TODO` (строго по порядку T2→T6), ставит `IN PROGRESS`, по завершении — `REVIEW` + запись в журнал `AGENT_LOG.md`: что сделал, что выяснил нового (обновить «Рабочие заметки», если исходник разошёлся с ожиданиями), какие решения принял сам и почему.
- **Ревьюер** (отдельная сессия): проверяет по «Приёмке» задачи, пишет вердикт в журнал; `DONE` или возврат в `IN PROGRESS` со списком замечаний.
- **Правила для всех**: не редактировать `original/` и `handoffs/`; не переименовывать публичные/используемые в XAML сущности; экономить кредиты — точечные чтения по METHOD_MAP вместо перечитывания исходника; писать файлы крупными Write-блоками; вопросы пользователю — только если блокер.
- **Git (требование пользователя)**: ВСЕ изменения отражаются НАПРЯМУЮ в ветке `main` — коммитить и пушить в `origin/main`, без PR и без фиче-веток. Если работа случайно ушла в другую ветку — немедленно смержить её в `main` и запушить.
- **Компиляция**: в песочнице нет C#-компилятора. Финальную сборку на `net10.0-windows` и ручные тесты делает пользователь ПОСЛЕ T6.
