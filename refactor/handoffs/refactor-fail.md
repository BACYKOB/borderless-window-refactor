# HANDOFF — рефакторинг BorderlessWindow на 7 partial + 2 фикса (июль 2026) — ИТОГ: РЕГРЕССИЯ

> Прочитать ПЕРЕД любой новой попыткой рефакторинга или фиксов ghost/gap.
> Это пост-мортем мульти-агентного рефакторинга (T1–T6, модели Opus/Fable/GPT).
> **Вердикт пользователя после сборки и ручных тестов: итоговая реализация ХУЖЕ исходного состояния.**
> Исходник-эталон: `../original/BorderlessWindow.cs` (5546 строк). Результат: `../Controls/*.cs` (7 partial-файлов).
> Сопутствующие документы: `../PLAN.md`, `../METHOD_MAP.md`, `../TASKS.md`, `../AGENT_LOG.md`, `../IDEAS.md`.

## TL;DR

- Класс `BorderlessWindow : Window` (namespace `ControlPanel`) разбит на 7 partial-файлов; удалены ControlzEx-путь, вся диагностика (TsLog/SnapLog/GripLog), мёртвые эксперименты; внедрены 2 новых фикса за const-флагами (`EnableSeamGapFix`, `EnableLeftEdgeGhostGuard`, оба по умолчанию `true`).
- Статическая верификация была безупречной (полнота 302 KEEP-членов, 0 дублей, 0 запрещённых токенов, баланс скобок 7/7 файлов) — **и это НЕ спасло**: все реальные проблемы были поведенческими, невидимыми без запуска.
- **Результат ручных тестов пользователя (Win11, 4K @150%, net10.0-windows)**:
  1. Фикс призрака (T3) — призрак НЕ убран.
  2. Фикс зазора (T4) — зазор НЕ убран.
  3. НОВЫЙ артефакт: **белая полоса слева при запуске приложения** (вероятный виновник — 1px левый NC-inset `EnableLeftEdgeGhostGuard` в `ThemedFrameNcCalcSize`; полоса — непокрашенная NC-область до первого прохода темы/DWM).
  4. **Удаление ControlzEx ухудшило поведение окна** — собственный chrome (перенесённый `WM_NCCALCSIZE`/`WM_NCHITTEST` путь) оказался хуже связки с ControlzEx, работавшей в исходнике.
  5. При обратном включении ControlzEx поверх нового кода: призрак при обычном ресайзе вернулся, остаточные белые вспышки, **стягивание за шапку из полноэкранного режима не работает вообще** (unsnap-restore цепочка `MaybeSuppressUnsnapResnapFrame` → WinEvent → `FinishUnsnapWinEventRestore`, видимо, конфликтует с ControlzEx-обработкой сообщений либо сломана переносом — не диагностировано).
- **Оценка пользователя: ухудшение первоначального состояния.**

### Удалено при переносе

- Весь ControlzEx-путь (`WindowChromeBehavior`, `AttachControlzExChrome`) — **решение оказалось ошибочным, см. итог**.
- Вся диагностика: TsLog/SnapLog/GripLog + троттлинг-поля (сотни call-site'ов).
- Мёртвые эксперименты: `EnableDividerDeferredResize`/`FrameSync`/`NoCopyBits`/`SingleBatch`, DivGuide, `TryBuildAlignedValidRects`, `GetWndClass`, старая серая маска `StartMaskCrossfade`/`MaskFadeLoop`/`Ease`, caption-watchdog.

### Фикс 1 — `EnableSeamGapFix` (зазор при joint-resize) — НЕ СРАБОТАЛ

- DivGrip-путь: вместо двухпроходного `ApplyDividerBatch` — предвычисление клампа соседа через `WM_GETMINMAXINFO` (`TryGetMinTrackSize` + кэш `_divMinTrackCache` на время драга) и ОДИН атомарный `BeginDefer/EndDeferWindowPos`.
- Frame-путь: «grower-first» pre-pass `FollowFrameResizeNeighborsPre` в `WM_WINDOWPOSCHANGING` (инвариант «перекрытие, не зазор»).
- Fallback при `false` — старая двухпроходная схема 1:1.
- **Итог: зазор остался.** Причина не диагностирована (нет логов — диагностика была удалена ПОЛНОСТЬЮ, отлаживать нечем).

### Фикс 2 — `EnableLeftEdgeGhostGuard` (призрак слева/сверху) — НЕ СРАБОТАЛ + ДАЛ АРТЕФАКТ

- 1px NC-inset слева в `ThemedFrameNcCalcSize`, симметрично верхнему `ThemedFrameInset`; `BorderThickness.Left` оставлен = 1 (правка аудитора: при 0 floating-окно теряло левую линию, т.к. DWM-полоса floating = `COLOR_NONE`).
- **Итог: призрак остался, плюс белая полоса слева при запуске.** Симметрия с верхним guard'ом не воспроизвела его эффект — вероятно, верхний inset «работал» в исходнике в сочетании с ControlzEx, а не сам по себе, либо его вклад в подавление призрака вообще был переоценён.

## УРОКИ — что пошло не так (главное в этом документе)

1. **Статическая верификация ≠ работоспособность.** Шесть задач, три аудита, независимые python-сканы — всё сошлось идеально, а поведение сломано. Для WM-message-level кода (NCCALCSIZE, WINDOWPOSCHANGING, WinEvent-hooks) НЕ существует статической приёмки. Любой будущий рефакторинг такого кода обязан идти шагами «изменение → сборка → ручной тест», а не «всё сразу → тест в конце».
2. **Big-bang запрещён.** Разбиение на 7 файлов + удаление ControlzEx + удаление всей диагностики + 2 новых фикса — В ОДНОЙ поставке. Когда всё сломалось, невозможно отделить: (а) ошибки переноса, (б) эффект удаления ControlzEx, (в) эффект самих фиксов. Правильно: сначала перенос 1:1 БЕЗ каких-либо удалений и фиксов → сборка → полный ручной прогон → только потом по одному изменению за итерацию.
3. **ControlzEx не был мёртвым кодом.** Handoff `resize-ghost.md` называл путь «ВАРИАНТ 1: убрать ControlzEx» экспериментальным направлением, а PLAN превратил его в решённое удаление. Фактически исходник работал ИМЕННО в связке с ControlzEx, и собственный chrome эту связку не воспроизвёл. Урок: «пользователь одобрил план» ≠ «предпосылка плана доказана». Гипотезы удалять из кода нельзя — только выключать флагом с сохранением рабочего пути.
4. **Полное удаление диагностики — до подтверждения работоспособности — ошибка.** TsLog/SnapLog были единственным инструментом диагностики зазора/призрака (логи в `../logs/` сделаны именно ими). Теперь регрессии нечем отлаживать. Урок: диагностику удалять ПОСЛЕДНЕЙ, после приёмки поведения, либо оставлять dormant за флагом.
5. **Фиксы «по симметрии» не работают без понимания механизма.** Фикс 2 зеркалил верхний inset, не имея доказательства, что верхний inset — причина отсутствия верхнего призрака. Aудит T3 поймал ложную предпосылку про `BorderThickness.Left=0`, но не поймал главную — что сам inset может не работать.
6. **Отсутствие компилятора в песочнице — известное ограничение, которое проигнорировали по сути.** «Компиляцию проверит пользователь после T6» превратило пользователя в единственный CI на самом последнем шаге. Минимальный тестовый csproj (идея из IDEAS §6) должен был появиться ДО T2, а не остаться идеей.
7. **Unsnap-цепочка хрупкая.** Перенос «байт-в-байт» Variant A/A+/A++ всё равно дал неработающее стягивание из fullscreen (при включённом ControlzEx). WinEvent-hook + message-suppression завязаны на точный порядок обработки сообщений — любое изменение соседнего кода (chrome, диспетчер WindowProc) может её ломать молча.

## Что делать следующему агенту

1. **Отправная точка — `../original/BorderlessWindow.cs`, НЕ `../Controls/`.** Исходник — последнее подтверждённо-рабочее состояние (со своими известными багами: призрак лево/верх, зазор joint-resize, вспышки).
2. Если разбиение на partial всё же нужно — делать его строго 1:1 (включая ControlzEx и ВСЮ диагностику), собрать, прогнать все 8 ручных сценариев (см. IDEAS §4), и только после приёмки — следующая итерация.
3. Ghost/gap-фиксы — только по одному, за флагом, с включённой диагностикой, с ручным тестом после каждого. Доказанные тупики — в `resize-ghost.md` и разделе «Чего НЕ делать» в `AGENT_LOG.md` — по-прежнему в силе.
4. Белая полоса слева при старте: первый подозреваемый — левый NC-inset фикса 2; проверить `EnableLeftEdgeGhostGuard=false` до любых других действий.
5. Неработающий unsnap из fullscreen при ControlzEx: диагностировать порядок обработки `WM_WINDOWPOSCHANGING` между `WindowChromeBehavior` и `MaybeSuppressUnsnapResnapFrame` — в исходнике они сосуществовали, в новом коде диспетчер `WindowProc` переписан.
